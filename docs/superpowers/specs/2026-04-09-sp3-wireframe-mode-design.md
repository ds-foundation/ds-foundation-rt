# SP3 — Wireframe Mode Design Spec

**Project:** DS v1 — Unified Design System  
**Sub-project:** SP3 — Wireframe Mode  
**Date:** 2026-04-09  
**Status:** Approved — ready for implementation plan

---

## Goal

Give non-designers (PMs, engineers) a way to rapidly prototype hi-fi ideas that are customer-testable but clearly not shippable. The mode applies a brand-neutral palette over the existing design system — same components, same layout, same spacing, different color tokens.

---

## Scope

1. **Repo reorganisation** — separate `components/` and `styles/` within `packages/react/src/`; clean `package.json` exports
2. **Wireframe CSS theme** — `semantic.wireframe.css` with `[data-theme="wireframe"]` overrides
3. **`sketch:` Tailwind variant** — escape hatch for one-off wireframe-specific class overrides
4. **`DesignSystemProvider`** — React context owning all theme state (light / dark / wireframe)
5. **`ThemeToggle`** — ready-made toggle button; no configuration required
6. **`useTheme()` hook** — for custom toggle UIs
7. **Changesets + CHANGELOG.md** — monorepo-wide changelog infrastructure

---

## Non-Goals

- Per-component `wireframe` prop — not needed; CSS token override covers all components
- Lo-fi / sketch aesthetic (Mac System 1 style) — this is hi-fi, brand-neutral, customer-testable
- Removing feedback color semantics — error/warning/success/info states are retained in wireframe mode
- Automated CI release pipeline — Changesets workflow is developer-run, not automated in SP3

### `package.json` exports

CSS is compiled by the Tailwind CLI build step (`tailwindcss -i src/styles/styles.css -o dist/styles.css`) — the export points to the compiled output:

```json
{
  "exports": {
    ".": {
      "import": { "types": "./dist/index.d.ts", "default": "./dist/index.js" },
      "require": { "types": "./dist/index.d.cts", "default": "./dist/index.cjs" }
    },
    "./styles.css": "./dist/styles.css"
  }
}
```

The `./styles.css` entry already exists in the package; no new export entry is needed — only the build script input path changes when CSS files move to `src/styles/`.

Consumer install is:

```ts
import { DesignSystemProvider, Button } from '@ds-foundation/react'
import '@ds-foundation/react/styles.css'
```

That is the entire setup. No additional config files, no token imports, no Tailwind setup required by the consumer.

### Migration: `src/*.tsx` → `src/components/*.tsx`

All 59 existing component files move from `packages/react/src/` to `packages/react/src/components/`. Internal imports between components use relative paths and must be updated. `index.ts` re-exports remain unchanged in signature — only paths change.

All CSS files move from `packages/react/src/` to `packages/react/src/styles/`. The `@import` paths in `styles.css` are updated accordingly.

## Section 2: Wireframe CSS Theme

### File: `packages/react/src/styles/semantic.wireframe.css`

```css
[data-theme="wireframe"] {
  /* Surfaces */
  --ds-bg:         #f5f5f5;
  --ds-surface:    #ffffff;
  --ds-surface-up: #ebebeb;
  --ds-overlay:    rgba(0, 0, 0, 0.4);
  --ds-sunken:     #e0e0e0;

  /* Text */
  --ds-text:          #1a1a1a;
  --ds-text-muted:    #666666;
  --ds-text-disabled: #aaaaaa;
  --ds-text-inverse:  #ffffff;

  /* Borders */
  --ds-border:       #cccccc;
  --ds-border-strong:#999999;
  --ds-border-focus: #4a7fa5;

  /* Brand (generic blue — not Ripple-branded) */
  --ds-primary:       #4a7fa5;
  --ds-primary-hover: #3a6f95;
  --ds-primary-fg:    #ffffff;
  --ds-primary-subtle:#ddeaf4;

  /* Feedback tokens are intentionally NOT overridden — they inherit from
     :root (light mode) and retain full semantic meaning in wireframe mode.
     Error/warning/success/info states must communicate correctly during
     customer testing sessions. */
}
```

Feedback tokens are not present in `semantic.wireframe.css`. They cascade from `:root` (light mode values in `semantic.css`) and are unchanged in wireframe mode. Error states, warnings, and success indicators must communicate meaning in customer-testing contexts.

### `styles.css` import order

```css
@import "@ds-foundation/tokens/build/css/variables.css";
@import "@ds-foundation/tokens/build/css/variables.dark.css";
@import "./semantic.css";
@import "./semantic.dark.css";
@import "./semantic.wireframe.css";   /* NEW */
@tailwind base;
@tailwind components;
@tailwind utilities;
```

### Activation

Applied to `<html>` by `DesignSystemProvider`:

```html
<html data-theme="wireframe">
```

Same mechanism as dark mode. No component changes required.

## Section 3: `sketch:` Tailwind Variant

A single addition to `tailwind.config.ts`:

```ts
// In plugins or future: extendVariants — for now, add to variants config
// Using the arbitrary variant pattern:
// sketch: variant means "when data-theme=wireframe is on an ancestor"
```

Added to `tailwind.config.ts` under `theme.extend` or as a plugin:

```ts
plugins: [
  plugin(function ({ addVariant }) {
    addVariant('sketch', '[data-theme="wireframe"] &')
  }),
],
```

Usage:

```tsx
<div className="bg-ds-surface sketch:border-dashed sketch:border-2">
```

This is an escape hatch only — the token override handles 95% of cases. `sketch:` is for the rare case where a wireframe-specific structural class is needed.

## Section 4: DesignSystemProvider

### File: `packages/react/src/components/DesignSystemProvider.tsx`

```tsx
type Theme = 'light' | 'dark' | 'wireframe'

interface DesignSystemProviderProps {
  children: React.ReactNode
  defaultTheme?: Theme  // default: 'light'
}
```

**Behaviour:**
- On mount: reads `localStorage.getItem('ds-theme')` — if present and valid, uses it; otherwise uses `defaultTheme`
- Writes `data-theme` attribute to `document.documentElement` on mount and on every theme change
- Removes `data-theme` when theme is `'light'` (light is the default/base, no attribute needed)
- Exposes theme state via `ThemeContext`

**`useTheme()` hook:**

```tsx
export function useTheme(): { theme: Theme; setTheme: (t: Theme) => void }
```

Throws if called outside `DesignSystemProvider`.

## Section 5: ThemeToggle

### File: `packages/react/src/components/ThemeToggle.tsx`

```tsx
export function ThemeToggle(): React.ReactElement
```

No props. Reads `useTheme()` internally.

**Behaviour:** cycles `light → dark → wireframe → light` on click.

**Render:**

```tsx
<Button variant="ghost" size="icon" aria-label={`Switch to ${nextTheme} theme`}>
  {theme === 'light'     && <Sun className="w-4 h-4" />}
  {theme === 'dark'      && <Moon className="w-4 h-4" />}
  {theme === 'wireframe' && <PenLine className="w-4 h-4" />}
</Button>
```

Uses the existing `Button` component. Uses Lucide icons (`Sun`, `Moon`, `PenLine`) already available in the package.

## Success Criteria

1. `npm install @ds-foundation/react` + one `import '@ds-foundation/react/styles.css'` is the entire consumer setup
2. `<DesignSystemProvider defaultTheme="wireframe">` applies wireframe tokens to all components with no component changes
3. `<ThemeToggle />` cycles correctly through all three themes
4. `data-theme="wireframe"` on `<html>` produces a visually distinct, customer-testable UI with neutral brand palette
5. Feedback states (error/warning/success/info) are visually unchanged in wireframe mode
6. `sketch:` variant resolves correctly in Tailwind output
7. `npx changeset add` records a changeset; `npx changeset version` bumps versions and updates CHANGELOG.md files
8. TypeScript compiles with zero errors (`tsc --noEmit`)
9. Docs app builds without errors
